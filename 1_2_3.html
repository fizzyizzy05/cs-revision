Why might projects fail?

  • Lack of research.
  • Poor project leadership.
  • Not enough time to complete the project.
  • Poor testing and quality control.

How would we gather the requirements of the user/client?

  • Questionnaires: these are a set of questions that are given out to potential users for them to complete.
  • Interviews: these are face-to-face discussions between the developers and potential users. These will often entail asking questions to the users about the projects.
  • Observations: this is where the developers will observe users of the existing system to understand how it works.
  • Meetings: this is where a group of users sit down with an analyst to discuss the current system.
  • Document analysis: this is where existing documentation in the current system is explained.

Different stages of development

Analysis

  • What is the problem? What does the software need to accomplish to overcome it?
  • A problem needs to be defined before it can be solved and the requirements established.
  • There will need to be the consideration of many different factors. In the example of a data processing system, this could include the data, procedures, future and the problems with the current system.

Design

  • What will the system look like? what will it be made of?
  • Depending on the type of project, the systems designer may consider any of the following:
    • Processing
    • Data structures
    • Output/Input
    • User interface
    • Security
    • Hardware

Programming and testing

  • This is the part where the software is developed and checked to see if it works.
  • A system needs to be thoroughly tested before installation to ensure that all errors are discovered and corrected.
  • Programmers will often come up with a test plan, and testing their own code is natural - it usually runs in parallel with the creation of code.

Installation

  • This is the stage at which the software will be installed where it needs to be.
  • Coding and testing will be carried out, and errors will be traced and fixed.
  • When everything is considered satisfactory, the software will be installed on the user's machine.

Evaluation

  • This is where the development of the new software is evaluated to see if it's effective at it's job.
  • This will come in the form of a post-implementation review, which will focus on the following:
    • A comparison between the old system and the new one.
    • An assessment of each aspect of the system against pre-set criteria.
    • What errors were made durng the development phase.
    • Unexpected advantages and issues.

Methodologies

Methodologies are a structured and agreed way of doing something. In the context of software development, it's a framework used to structure, plan and control the process of developing a solution.

Waterfall Model

  • Each step is completed one at a time, contigously. Each step has specific outputs that lead smoothly into the next step.
  • It's possible to go back to a previous stage, but developers will need to work back down through the stages.
  • The client is inolved at the beginning, but from there on has little input until the evaluation stage.
  • The model is adopted from the manufacturing industry - it makes sense there since hardware made later in the project had high cost implications, so getting each stage right was important. But in software development, it has been superseded by more effective models.

Agile Model

  • Agile is an iterative process heavily based on customer involvement throughout the development process, making use of small multitasking teams of programmers.
  • The process starts with the creation of a prototype, which the customer gives feedback on.
  • This model is suited to small groups of devs working together as opposed to large projects with developers working on seperate parallel parts.
  • The methodology is reliant on face-to-face communication.
  • At each stage, the a prototype is built with client participation to ensure the resulting system is built in line with their needs.
  • RAD and XP development (explained below) are based on Agile.

Rapid Application Developmnent (RAD)

  • Some very large projects may be developed over a long period of time - technology and user requirements will change throughout this.
  • RAD involves workshops and focus groups as a way to gather requirements instead of a formal requirement document.
  • Prototyping is used to refine the system in response to user involvement and feedback.
  • Each part of the system is developed within a strict time limit - the outcome will still be good enough, even if it isn't perfect.
  • If software components can be re-used, they likely will be.

Extreme Programming (XP)

  • Software following the XP model is developed by a team of less than a dozen people working in a communal work space.
  • Developers may freely share ideas and help each other in developmnent. In XP, code quality is important, and paired programming helps with this.
  • The software is developed incrementally - this means that new expanded versions of the software package will appear on a regular basis.
  • XP does have the issue of focusing on code rather than design, however.

Spiral Model

  • The spiral model uses structured steps but has the software developed in iterative stages.
  • At the start of the process, the requirements are defined and developers will work towards an initial prototype.
  • The cycle will continue until a refined product is generated.
  • Each time around the spiral, the following activities are done:
    • Analyse the requirements for the next prototype.
    • Design the next version.
    • Implement (code and test) the new prototype.
    • Evaluate the new prototype, generating a plan for the next generation.
